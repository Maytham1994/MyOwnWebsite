Shifty (4th Year Project)

Shifty: Automatic Bicycle Transmission System

Project Description

The shifty is a smart device which accompanies the cyclist on their journey through the urban landscape.

The automatic transmission module built into the shifty is designed to take the task of shifting gears off the hands of the cyclist. The shifty anticipates and reacts to environmental changes to keep cycling as comfortable and efficient as possible.

The project consists of three microcontrollers;

  • UI/GPS Controller
    • User Interface using an LCD
    • SD Card for logging and map image file storage
    • GPS module for coordinates and GPS time
    • SPI connection to Algorithm/Bluetooth Controller
  • Algorithm/Bluetooth Controller
    • Transmission Algorithm
    • Bluetooth Connection to Sensors (Speed/Cadence Sensor, and Heart Rate Sensor)
    • Connection to Orientation Sensors
    • I2C Connection to Motor Controller
  • Motor Controller
    • Motor Driver
    • Motor Feedback and current detection

The group consists of 4 Electrical Engineering students, the work is divided between all members with my roles identified below;

  • Team Lead
    • Lead meetings and assign deadlines that fit with the course deadlines
    • Schedule meetings
    • Look for and ensure external funding is applied for and received to keep the project development going
  • UI Subsystem Designer
    • Design a simple, easy to use UI running on an STM32F469 and a DSI LCD touchscreen
    • Integrate an SD card for map tile images and also for logging of the system
      • Custom log formating that includes time, message and data in a comma separated value (csv) format
    • Integrate a GPS module as a separate thread to obtain GPS coordinates
      • Thread waits for a message from the main UI thread in order to retrieve the data from the GPS module
      • GPS coordinates are used in calculating map tile values
      • UI thread uses those map tile values to retrieve the map tiles from the SD card and display on the screen
      • The map is based on open source software by calculating tile numbers and stitch together the required map
    • Integrate SPI drivers to communicate with a algorithm/bluetooth controller
      • Thread waits for a message from the main UI thread in order to retrieve data from or send data to the 2nd controller
      • Data retrieved from the 2nd controller can include speed, rpm, battery status, heart rate, and other data bluetooth data
      • Data sent to the 2nd controller is settings related to the bicycle that the user inputs in the UI
      • 2nd controller also passes logs through the SPI in order to be included in the logs on the SD card
      • External interrupts are used to allow the 2nd controller to notify the UI controller of data
    • Messaging queues are used in order to coordinate and send messages and data between the different threads
    • Common code and resources (such as the SD card) are locked with semaphores to ensure no conflicts between the threads
    • Keil CMSIS-RTX RTOS is used for threading, message queues and semaphores
  • Hardware Reviewer
    • Review the hardware designs before they are sent out for printing

CanSat 2016

CanSat 2016

Project Description

The Carleton University Raven Knights CanSat team competes every year in the International CanSat Competition in Abilene, Texas.

The purpose was to design a rocket deployed glider that will be launched to approximately 1000 meters, and must descent with a circular radiuss of 1 km. Additional telemetry data of atmospheric conditions was also calculated, managed, saved and transmitted to a ground station for analysis.

The competition was organized by esteemed institutions including:

  • National Aeronautics and Space Administration
  • American Astronautical Society
  • American Institude of Aeronautics and Astronautics

My role was to help with the Software of the Glider, help with the ground station data dumping software, used in post flight EEPROM dumping, design the container hardware and software used for the purpose of releasing the glider from the container. My role included;

  • Design container hardware
  • Design the I2C driver (glider/container)
  • Collect Pressure and Temperature from BMP180 sensor (glider/container)
  • Calculate the Altitude using the Pressure data
  • Maintain the flight time using an RTC (DS1338) (glider/container)
  • Maintain all the telemtry data and store them in a AT24CM01 EEPROM (glider/container)
  • Design the release mechanism using a resistor and fishing wire by applying large current to a small resistor and giving it more than the rated power, the resistor desipates lots of heat which inturn burns the fishing wire
  • Design the main software thread of execution for the container to perform the necessary tasks
  • Implement bug fixes in the glider's main software that helped in improving camera usage, data transmission, and image transmission to the ground station

Our team achieved 2nd place out of 70+ registered teams.

RoboVoy (Autonomous Vehicle)

Project Description

As the third year Electrical Engineering project, my team decided to design a somewhat autonomous vehicle. Using a JeeNode, an Ultrasonic Range Finder, two IR Range Finders, I/O Expander, Car Chassis and DC Motor Driver the project idea came to reality.

The purpose of the car is to autonomously follow the vehicles in front of it. This is somewhat autonomous since it can only follow other vehicles. The idea has the potential of being taken and improved with full autonomous movement.

Safety features such as remote activation were also implemented in case autonomous movement is lost.

The use of I2C, RF and Pulse Width Modulation was used in this project.
